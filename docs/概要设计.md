### 轻量级域名管理系统概要设计文档

#### 1. 前后端技术选型

为了实现极简部署和低资源占用，使用以下技术栈组合：

* **后端 (Backend): Go (Golang) + Gin / Fiber**
* **优势**: Go 语言编译后生成独立且无外部依赖的二进制文件，非常适合部署在路由器、NAS 或低配 VPS 上。内存占用极低，且天生支持高并发，非常适合处理 DDNS 的后台轮询任务。
* **替代方案**: Node.js (Express/NestJS) 或 Python (FastAPI)。虽然开发迅速，但需要依赖相应的运行时环境（Node 模块或 Python 虚拟环境），略微违背“极简依赖”的初衷。


* **前端 (Frontend): Vue 3 (Composition API) + Vite + Tailwind CSS (或 Element Plus)**
* **优势**: 编译后为纯静态的 HTML/CSS/JS 文件，可以直接交由 Go 后端统一作为静态资源托管，无需额外部署 Nginx，实现真正的“单文件启动”。


* **数据库 (Database): SQLite 3**
* **优势**: 单文件关系型数据库，无需额外部署 MySQL 或 PostgreSQL 服务进程。完全融入应用代码中，数据备份只需复制一个 `.db` 文件。



#### 2. 系统架构与模块设计

系统采用标准的 B/S (Browser/Server) 架构。前端负责用户交互与配置下发，后端负责与第三方 DNS 供应商（阿里云、Cloudflare）的 API 交互以及执行后台定时任务。

核心模块划分为以下几个部分：

* **供应商管理模块 (Vendor Manager)**
* 支持添加、修改、删除供应商账号如阿里云 (AccessKey/SecretKey), Cloudflare (API Token/Global API Key) 。
* 供应商支持: 阿里云, 腾讯云, 华为云, cloudflare, aws
* 支持多账号管理，为每个账号分配内部 ID。


* **域名与解析管理模块 (Domain & Record Manager)**
* **域名同步**: 根据配置的供应商账号，调用相应 API 拉取主域名列表并缓存入库。
* **静态解析**: 提供图形化界面，实现对 A, AAAA, CNAME, TXT, MX 等记录的增删改查。后端负责将实时调用对应供应商接口。


* **公网 IP 检测模块 (IP Checker)**
* **获取策略**: 支持通过公共接口（如 `api.ipify.org`, `checkip.amazonaws.com`）获取当前主机的公网 IPv4/IPv6。


* **动态域名解析模块 (DDNS Worker)**
* 用户可将特定的子域名绑定至 DDNS 任务。
* 后台使用协程 (Goroutine) 启动定时轮询任务（如每 5 分钟执行一次）。
* 比对**当前公网 IP** 与**记录的上一次 IP**，若发生变化，则调用供应商 API 更新 DNS 记录，并记录日志。



#### 3. 数据库结构设计 (SQLite)

数据库表设计应尽量精简，核心表结构如下：

* `vendors` (供应商表)
* `id`: 主键
* `name`: 别名 (如 "我的阿里云")
* `provider`: 类型 (`aliyun` | `cloudflare`)
* `api_key`: 凭证 ID / Token
* `api_secret`: 凭证密钥 (加密存储)


* `domains` (域名表)
* `id`: 主键
* `vendor_id`: 关联供应商
* `domain_name`: 主域名 (如 `example.com`)


* `records` (解析记录缓存表 - 可选，用于加速前端展示)
* `id`: 主键
* `domain_id`: 关联域名
* `record_id`: 供应商端的记录 ID
* `host`: 主机记录 (如 `www`)
* `type`: 记录类型
* `value`: 记录值


* `ddns_tasks` (DDNS 任务表)
* `id`: 主键
* `domain_id`: 关联域名
* `sub_domain`: 需要动态更新的子域名
* `record_type`: A 或 AAAA
* `check_method`: IP 获取方式 (API / 网卡)
* `last_ip`: 最后一次成功更新的 IP 地址
* `status`: 运行状态 (运行中/暂停)



#### 4. 核心业务流程

**DDNS 自动更新流程：**

1. 定时器触发 DDNS 检查任务。
2. 遍历 `ddns_tasks` 表中所有状态为“运行中”的任务。
3. 根据任务配置的 `check_method` 获取当前主机的真实公网 IP。
4. 对比获取到的 IP 与 `last_ip`：
* **一致**: 跳过，等待下次轮询。
* **不一致**:
* 查询该子域名在供应商处的 `record_id`。
* 调用阿里云/Cloudflare API 更新该记录的 IP 值。
* 更新成功后，将新 IP 写入 `ddns_tasks` 表的 `last_ip` 字段，并记录成功日志。





#### 5. 部署方案设计

考虑到轻量化，推荐最终交付形态为**单二进制文件**或**极简 Docker 镜像**：

* **编译阶段**: 前端使用 Vite 编译出静态文件 (`dist` 目录)，后端 Go 程序使用 `go:embed` 将前端 `dist` 目录打包进二进制文件中。
* **运行阶段**: 只需要执行编译出的 `app.exe` 或 `./app`，程序会自动在同级目录生成 `data.db` (SQLite 数据库) 和 `config.yaml` (端口等基础配置)，并同时提供前端页面和后端 API 服务。
* **Docker 镜像**: 基于 `alpine` 或 `scratch` 构建镜像，体积可控制在 20MB 以内。
