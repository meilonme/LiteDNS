# LiteDNS 详细设计 - 10 系统架构与流程

## 1. 系统架构概览
LiteDNS 采用单体后端 + 静态前端的 B/S 架构。前端由后端统一托管，后端内部以模块分层组织。

## 2. 模块划分
### 2.1 Auth 模块
- 管理员登录、登出、改密、会话续期、会话撤销。
- 首次启动生成随机密码，首次登录强制改密。

### 2.2 Vendor 模块
- 供应商账号 CRUD。
- 凭证加解密与连通性验证。

### 2.3 Provider Adapter 模块
- 定义统一 Provider 接口，屏蔽厂商差异。
- V1 实现：`aliyun`、`cloudflare`。

### 2.4 Sync 模块
- 域名与记录查询时执行缓存过期判定。
- 过期则调用 Provider 拉取最新数据并更新本地缓存。

### 2.5 Record 模块
- 提供解析记录读写接口。
- 写入流程：远端成功 -> 本地缓存回写 -> 系统日志写入（`type=operation`）。

### 2.6 DDNS 模块
- 任务管理（创建、暂停、恢复、删除、手动执行）。
- 调度循环与任务执行器。
- 执行日志记录与失败计数管理。

### 2.7 Log 模块
- 统一系统日志查询（按 `type` 区分 DDNS 任务、公网 IP 检查、操作日志）。
- 定时清理 90 天前历史日志。

### 2.8 Config 模块
- 加载 `config.yaml` + 环境变量覆盖。
- 提供系统配置查询与部分可热更新项。

## 3. 运行时组件
### 3.1 HTTP API Server
- 基于 `Gin` 提供 `/api/v1` 接口。
- 静态资源由同进程托管，服务后台管理页面。

### 3.2 Scheduler
- 单实例内存调度器，每 1 秒 Tick 扫描 `next_run_at <= now` 的运行中任务。
- 公网 IP 检查器按 `ip_check_interval_sec` 周期执行，将结果写入 `public_ip_check_settings.public_ip`。
- 仅单实例场景，不实现分布式抢占锁。

### 3.3 Storage
- SQLite 作为主存储。
- 关键写路径采用事务保证一致性。

## 4. 关键时序流程
## 4.1 登录鉴权流程
1. 用户提交用户名与密码。
2. Auth 模块校验密码哈希。
3. 通过后签发会话 token，写入 `sessions`（仅哈希）。
4. 返回会话信息与 `must_change_password` 标记。
5. 若需改密，前端必须先调用改密接口后方可继续其他操作。

## 4.2 查询触发同步流程（阻塞返回最新）
1. 客户端请求域名或记录列表。
2. Sync 模块检查 `last_synced_at` 是否超过 `sync_ttl_sec`。
3. 若未过期：直接返回缓存。
4. 若过期：同步执行远端拉取（分页）并在事务内刷新本地缓存。
5. 同步完成后返回最新数据；若失败则返回错误，不覆盖旧缓存。

## 4.3 DDNS 轮询更新流程
1. Scheduler 扫描到可执行任务（`status=running` 且 `next_run_at <= now`）。
2. 执行器读取 `public_ip_check_settings.public_ip` 作为目标公网 IP。
3. 调用 Provider `ListRecords` 查询目标记录当前值。
4. 若供应商值与 `public_ip` 相同：结束本次执行（`skip`）。
5. 若不同：调用 Provider `UpsertRecord` 更新；若记录不存在则创建。
6. 成功后更新任务状态字段并写 `system_logs`（`type=ddns_task`）。

## 4.4 失败重试流程
1. 单次任务执行失败后：
- `consecutive_failures + 1`
- 写 `last_error`
- 写失败日志
- 计算下一次执行时间（维持任务基础周期）
2. 下次执行成功后清零 `consecutive_failures`。

## 4.5 日志清理流程
1. 每日固定时刻触发清理任务。
2. 删除 `created_at < now - 90d` 的 `system_logs`。
3. 清理结果写入系统日志。

## 5. 异常与降级策略
### 5.1 外部 IP 源不可用
- 公网 IP 检查器按顺序尝试所有配置源。
- 全部失败则保留旧 `public_ip`，写失败系统日志（`type=public_ip_check`），等待下一周期重试。

### 5.2 Provider API 限流或超时
- 单次任务失败后标记失败并等待下个周期。
- 不进行任务内重试，避免放大上游限流压力。

### 5.3 目标记录不存在
- 由 `UpsertRecord` 自动创建记录并继续流程。
- 创建后同步回写 `remote_record_id` 到本地缓存。

### 5.4 部分写入失败
- 原则：远端写成功前不得更新本地最终状态。
- 本地事务失败时：保留失败日志并触发下一周期自愈同步。

### 5.5 配置错误
- 主密钥/数据库路径等启动关键项错误时直接启动失败。
- 非关键运行项异常（例如某 IP 源格式错误）记录告警并跳过该项。
- `public_ip_check` 关闭时禁止创建 DDNS 任务；存在 DDNS 任务时禁止关闭 `public_ip_check`。

## 6. 运行与部署
### 6.1 单二进制模式
- `go:embed` 打包前端静态资源。
- 启动后同时提供前端页面与 API。

### 6.2 配置加载优先级
1. 环境变量
2. `config.yaml`
3. 默认值

### 6.3 启动检查清单
- 主密钥格式与长度。
- SQLite 可读写。
- 管理员初始化状态。
- 任务调度器与日志清理器注册状态。

## 7. 调度与并发模型
- Scheduler 单 goroutine 负责选取到期任务。
- 任务执行采用有界 worker 池（默认并发数 4）。
- 同一 `ddns_task` 在任意时刻只允许一个执行实例。
- 并发冲突通过 `status + next_run_at + 乐观版本字段`（或等价行级条件更新）避免重复执行。

## 8. 流程验收标准
- 登录后必须遵循首次改密约束。
- 查询过期缓存时返回数据必须来自本次同步后的结果。
- DDNS 在 IP 不变时不调用 Provider 写接口。
- DDNS 在记录缺失时可自动创建并持续运行。
- 连续失败和恢复后的状态字段变化与日志记录保持一致。
